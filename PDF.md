## 3. Implementation Summary - User Registration

### Registration fields used

Based on `backend/src/main/java/edu/cit/racaza/annimemo/auth/dto/RegisterRequest.java`:

- `username`
- `password`
- `firstName`
- `lastName`
- `email`

### Validation process

Validation is done through Jakarta Bean Validation annotations in `RegisterRequest`:

- `username`: required, 3-30 characters
- `password`: required, 12-100 characters, must contain uppercase, lowercase, number, special character, and no spaces
- `firstName`: required, max 50 characters
- `lastName`: required, max 50 characters
- `email`: required, valid email format, max 120 characters

The endpoint uses `@Valid` in `AuthController` (`POST /api/auth/register`) so invalid input is rejected before service logic continues.

### How duplicate accounts are prevented

From `AuthService.register(...)`:

- Username check: `existsByUsername(normalizedUsername)`
- Email check: `existsByEmail(normalizedEmail)` after trimming and converting email to lowercase

Database-level protection is also present in `PostgresEmailIndexInitializer`:

- Creates case-insensitive unique index:
	`CREATE UNIQUE INDEX IF NOT EXISTS uk_app_users_email_ci ON app_users (lower(email))`

This prevents duplicates like `User@Mail.com` vs `user@mail.com`.

### How passwords are stored securely

Passwords are never stored as plain text. In `AuthService.register(...)`:

- Password is hashed with `BCryptPasswordEncoder` (`passwordEncoder.encode(request.password())`)
- Stored in the `passwordHash` column of `app_users`

## 4. Implementation Summary - User Login

### Login credentials used

Based on `LoginRequest`:

- `username` field (accepts either username or email)
- `password`

### How the system verifies users

From `AuthService.login(...)`:

1. Reads login identifier from `request.username().trim()`
2. Looks up user by username first
3. If not found, looks up by lowercase email
4. Verifies password using `BCryptPasswordEncoder.matches(...)`
5. If lookup or password check fails, throws `AuthException("Invalid username/email or password")`

### What happens after successful login

On success:

- A token is generated by `generateToken(...)` using `username + timestamp` encoded in Base64 URL format
- Backend returns `AuthResponse` with message, token, and username
- Frontend stores token in localStorage and navigates user to dashboard

## 5. Database Table

From `backend/src/main/java/edu/cit/racaza/annimemo/auth/model/AppUser.java`:

- Table: `app_users`
- Columns:
	- `id` (Long, primary key, auto-generated)
	- `username` (unique, required, length 30)
	- `passwordHash` (required, length 100)
	- `firstName` (required, length 50)
	- `lastName` (required, length 50)
	- `email` (unique, required, length 120)

Additional DB constraint:

- Case-insensitive unique index on email (`uk_app_users_email_ci` on `lower(email)`)
